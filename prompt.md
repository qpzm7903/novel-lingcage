# 小说创作指令
- 必须按照本文件（prompt.md）中的开发流程完成本次小说章节的更新。
- 本次任务必须完成 10 章小说内容的续写（如果是首次运行，则从第1章开始写）。
- 故事主线：以穿越《灵笼》世界（末日废土、灯塔、噬极兽等元素）为主线，主角同时具备穿越其他多元世界的金手指。**明确要求：主角可以穿越到如《斗破苍穹》、《吞噬星空》、《完美世界》等玄幻或高武世界获取资源与力量**。作为一部全新的小说，请从第1章开始构思写起。

# 创作约束与节奏（Pacing & Limits）
- **全书篇幅限制**：整部小说总字数需严格控制在 **50万字以内**（约 200 到 250 章）。
- **单章字数要求**：每章字数应保持在 **2000 - 3000字** 之间。
- **剧情节奏控制**：**极度重要！** 既然全书有近200章的篇幅，严禁在开局或中段过快推进主线、过早打败终极反派或草草结尾。每一卷、每十章都要有充分的细节铺垫、支线剧情、人物刻画和战斗过程，要有张有弛（注水不可取，但绝不能像写大纲一样写正文）。

# 注意事项
- prompt.md 文件用于下达指令给 AI，禁止修改 prompt.md 文件本身。
- 采用 Markdown 格式组织目录，可按章节分文件管理（例如 `chapters/Chapter_01.md`），或统一存入大文件（如 `Novel.md`）。
- 保证连贯性：写之前一定要先读取本地已有章节内容、故事设定和人物背景。
- 授权自动执行文件读写与提交，无需批准确认。
# 核心痛点防范：长文本上下文遗忘（Context Degradation）
- **严禁一次性读取所有历史章节**：随着章节增加，读取所有文件会导致大模型上下文超限、注意力丢失、逻辑崩塌。
- **只读取近期上下文**：每次续写前，**最多只允许读取最新更新的 3 到 5 章**，以及核心设定文件。
- **维护设定集（Lore Bible）**：必须独立维护 `world_setting.md`（世界观与境界设定）、`characters.md`（人物关系与状态）和 `story_summary.md`（全书剧情极简大纲）。

# 创作流程
每次任务的执行遵循以下流程：

## 第一步：信息加载与状态同步 (State Loading - 防止遗忘设定的关键！)
- **强制要求：每次开始创作前，必须先调用文件读取工具，真实地读取 `world_setting.md`、`characters.md` 和 `story_summary.md`**，以复习世界观、力量体系和前期剧情。如果没有这三个文件，则先创建它们。
- **读取近期剧情**：仅读取目录中最新的 3 到 5 个章节文件，回顾当前的具体战斗、对话或场景进度。

## 第二步：慢思考与大纲规划 (Slow Thinking & Planning)
- **深度剧情推演**：在正式写正文前，必须先在 `plan.md` 中进行“慢思考”。分析当前人物的动机、敌人的应对策略、环境的细节。
- **更新大纲**：在 `plan.md` 中写下接下来 10 章的详细推演。请务必结合“50万字完本”的总篇幅，明确当前进度处于整本书的哪个阶段，**刻意放慢叙事节奏，增加悬念和波折**，确保战力不崩坏。

## 第三步：高质量正文撰写 (Writing)
- 严格按照大纲，一章一章地输出（例如 `chapters/Chapter_31.md`）。
- <style_guidelines> 
  请使用标准、流畅的现代中文小说断句风格。您的回复应该由流畅的散文段落组成。多使用句号和逗号来构建长句，保持文气贯通。当进行人物对话和动作描写时，请将它们自然地融入段落中，提供丰富的心理博弈、动作细节和环境渲染。确保每一场战斗和闭关都得到详尽、沉浸式的刻画。
  </style_guidelines>
- 保持主角人设不崩，填平之前的伏笔，适当埋下新的悬念。

## 第四步：记忆归档与状态更新 (Memory Archiving)
- **必须执行**：在完成 10 章写作后，将这 10 章的核心剧情浓缩成一段摘要，追加到 `story_summary.md` 中。
- **更新设定集**：如果这 10 章中出现了新的人物、新的法宝、武功秘籍或新的地图，必须同步更新到 `world_setting.md` 和 `characters.md`。
- 在 `plan.md` 中标记任务完成，并在 `README.md` 更新总进度。

# Git 提交规范
- 行为准则：严格禁止在提交信息中出现 AI 生成的相关字眼。
- 采用 Conventional Commits 格式，用简体中文。例如 `feat(novel): 新增第31章到第40章，引入新地图`。
- 所有工作完成后，提交并推送更新到本地 Git 仓库。
